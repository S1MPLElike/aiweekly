# 冲击波智能工作周报系统

---

**冲击波智能工作周报系统**：基于 Go + Vue 3 + DeepSeek LLM 的工作记录管理平台，解决职场人每日汇报繁琐、报告撰写耗时的痛点，通过 AI 自动生成专业日报、周报。

---

## 核心价值

### 解决什么问题

1. **效率问题**：手动写日报/周报耗时 15-30 分钟/次，系统自动化生成
2. **规范问题**：报告格式不统一、内容不完整，AI 生成保证专业规范
3. **数据孤岛**：工作记录分散在各种文档中，通过结构化存储实现数据统一管理
4. **复用价值**：历史记录可追溯，方便复盘和查询

### 应用场景

- 实习生/职场新人：快速生成规范的工作汇报
- 项目管理：团队成员统一汇报格式
- 个人效率：减少事务性工作，聚焦核心任务

---

## 后端技术栈（详细）

### 技术选型

| 技术 | 说明 | 为什么选 |
|------|------|----------|
| **Go 1.25** | 高性能后端语言 | 编译型、并发强、学习曲线低 |
| **Gin 框架** | HTTP Web 框架 | 轻量、性能好、路由灵活 |
| **GORM** | ORM 库 | 结构化操作、避免 SQL 注入 |
| **MySQL** | 关系数据库 | 成熟稳定、团队熟悉 |
| **JWT** | Token 认证 | 无状态、可扩展 |

### 架构设计

```
用户请求 → Gin 路由 → JWT 中间件 → Handler → Service → MySQL
                                    ↓
                              LLM Service → DeepSeek API
```

### 核心模块

#### 1. 鉴权系统

- **双 Token 机制**：AccessToken（2小时）+ RefreshToken（7天）
- **TokenVersion**：支持 Token 主动失效（如修改密码后）
- **Gin 中间件**：统一拦截、全局鉴权
- **数据库查询优化**：减少 N+1 查询问题

#### 2. 业务接口

- **工作记录 CRUD**：按日期聚合、时间段校验
- **统计聚合**：按天/周/月计算工时、任务数
- **报告生成**：从工作记录 → 日报 → 周报的聚合链路

### 数据库设计

| 表 | 关系 | 设计亮点 |
|------|------|----------|
| 用户 | 主表 | 密码 bcrypt 加密、TokenVersion |
| work_records | user 1:N | 联合索引 (user_id, record_date) |
| daily_reports | user 1:N | 唯一索引 (user_id, report_date) |
| weekly_reports | user 1:N | 唯一索引 (user_id, week_start, week_end) |

## AI 接入

### 技术方案

```go
// LLM 服务封装
type Service struct {
    APIKey  string
    Client  *http.Client  // 60秒超时控制
}

// 统一调用 DeepSeek API
func (s *Service) CallDeepseek(messages []Message) (string, error)
```

### Prompt Engineering

#### 1. 日报生成

```markdown
【角色】专业日报助手

【输入】多条工作记录（标题、内容、时间段）

【输出要求】
1. JSON 格式（严格）
2. 保留所有关键信息
3. 不过度概括
4. 格式：{ title, content }
```

#### 2. 周报生成

```markdown
【角色】给领导看的周报助手

【输出结构】
一、本周工作概述（2-3句话）
二、主要工作成果（具体项目 + 成果）
三、工作亮点（最有价值的突破）

【JSON 格式】
{ title, summary, content }
```

### 工程化细节

1. **JSON 严格校验**：LLM 输出必须 JSON，解析失败则保存原始内容
2. **容错降级**：LLM 不可用时自动切换简单模板
3. **日志追踪**：记录请求/响应内容，便于排查
4. **超时控制**：60 秒超时，避免请求阻塞

## 技术难点

### 1. LLM 输出稳定性

**问题**：LLM 可能输出非 JSON 格式或 Markdown 包裹的 JSON

**解决**：
- 强制 System Prompt 要求只输出 JSON
- 捕获 JSON 解析错误，保存原始响应
- 降级方案保证功能可用

### 2. 报告聚合链路

**问题**：从工作记录 → 日报 → 周报的多层聚合

**链路**：
```
用户记录（多条）→ 日报生成（LLM）→ 保存 → 
周报查询（多天日报）→ 周报生成（LLM）→ 保存
```

### 3. JWT 鉴权设计

**问题**：Token 安全、续期、失效机制

**方案**：
- 双 Token + TokenVersion
- 中间件统一鉴权
- 支持 Cookie 和 Header 两种方式

### 4. 数据库查询优化

**问题**：按日期范围查询工作记录

**优化**：
- 联合索引 (user_id, record_date)
- 聚合查询计算统计
- 避免全表扫描
